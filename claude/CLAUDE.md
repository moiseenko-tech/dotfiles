# Global conventions

## Working style

**Never start writing or modifying code without explicit approval.**

When the user asks a question, describes a problem, or raises a topic related to development or bug fixes — stop, think, and discuss first. Do not jump to implementation. Instead:

1. Ask clarifying questions if the task is ambiguous.
2. Propose an approach or outline the changes you plan to make.
3. Wait for the user to explicitly say to proceed (e.g. "go ahead", "do it", "implement it").

Only begin writing code, editing files, or running commands when the user has clearly given the go-ahead.

### Development cycle

For any non-trivial task (more than one logical step):

1. **Plan first.** Before touching any code, present a numbered step-by-step plan. Wait for explicit approval before starting.
2. **One step at a time.** Execute exactly one step, then stop. Describe what was done and what comes next. Wait for the user's go-ahead before proceeding.
3. **Tests before implementation.** Writing tests and writing the implementation are always separate steps, in that order. Never bundle them into one step.
4. **Step size.** Steps should be balanced — small enough to review comfortably, large enough to be meaningful. A step is one logical unit of work (e.g. writing tests for a feature, adding a migration, wiring up a new adapter).

For trivial single-step tasks, no plan is required — but the go-ahead rule still applies.

## Code and documentation language

All code, comments, docstrings, and documentation must be written in American English.

## License

Every project must contain a `LICENSE` file.

- For open source projects: [MIT License](https://opensource.org/licenses/MIT) is the default choice.
- For proprietary/work projects: the license file must contain only the following copyright notice:

```
Copyright (c) <YEAR>, <Company Name>. All rights reserved.
```

If the project spans multiple years, use a range:

```
Copyright (c) 2024-2026, <Company Name>. All rights reserved.
```

## CHANGELOG

Every project must contain a `CHANGELOG.md` file following the [Keep a Changelog](https://keepachangelog.com/) format.

## Versioning

All projects follow [Semantic Versioning](https://semver.org/) (MAJOR.MINOR.PATCH).

## Commit style

All commits must follow the [Conventional Commits](https://www.conventionalcommits.org/) specification.

Examples: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `ci:`.

Commit footers include references to related tickets:

```
Refs: JIRA-1234, JIRA-1268
```

Do not add `Co-Authored-By` trailers to commit messages.

## Git workflow

- **Rebase over merge**: always rebase instead of merging to keep a clean linear history.
- **No pushing**: only the user pushes to the remote. Never run `git push`.

## Release workflow

Before each release, a dedicated **Release commit** is created that aggregates all release-finalization changes:

- Version bump in version-bearing config files (`pyproject.toml`, `go.mod`, etc.)
- `CHANGELOG.md` update
- Version bump in observability labels (e.g. `docker-compose` service labels)
- Lockfile update (`poetry.lock` / `uv.lock` / `go.sum`)
- Any other housekeeping required to finalize the release

The commit message format is exactly: `Release vX.Y.Z`

The version tag (`vX.Y.Z`) is placed on this Release commit — not on any other commit.

### Tag message

Annotated tag message format:

```
Release vX.Y.Z

<one or two sentences describing the essence of the release>

See CHANGELOG.md for full details.
```

### GitLab release notes

When preparing a release, generate the release notes as a temporary markdown file (e.g. `release-notes-vX.Y.Z.md`). The user will copy the contents manually into the GitLab release interface.

Format:

```markdown
## What's Changed

### Features
- feat: short description (JIRA-1234)

### Bug Fixes
- fix: short description (JIRA-1268)

### Breaking Changes
- ...

---
**Full Changelog**: CHANGELOG.md
```

Omit sections that have no entries. The `Breaking Changes` section is only included when applicable.

## Linting

- Linters are mandatory in every project.
- Linter configuration is the responsibility of the user — configs must be standardized and are not generated by Claude.
- Every project must contain a `Makefile` with targets for running linters locally during development.

## Application architecture

Backend services and APIs must follow the [Twelve-Factor App](https://12factor.net/) methodology. Key principles to enforce:

- **Config**: all configuration comes from environment variables, never from hardcoded values or config files committed to the repo.
- **Dependencies**: dependencies are explicitly declared (e.g. `pyproject.toml`, `go.mod`) and never rely on system-wide packages.
- **Backing services**: databases, queues, caches, and external APIs are treated as attached resources referenced via environment variables.
- **Logs**: applications write logs to stdout as an unbuffered event stream — never manage log files directly.
- **Dev/prod parity**: keep development, staging, and production environments as similar as possible.

This section does not apply to CLI tools, scripts, or linter configs.

## Python projects

- For new projects, prefer [uv](https://github.com/astral-sh/uv) as the package manager. Use [hatchling](https://hatch.pypa.io/) as the build backend.
- Linter: [ruff](https://github.com/astral-sh/ruff).
- Formatter: [black](https://github.com/psf/black).
- Type hints are mandatory throughout the codebase.
- Testing framework: [pytest](https://pytest.org/) (pure pytest, no additional test frameworks).
- Tests follow the **AAA pattern** (Arrange-Act-Assert): each test has three clearly separated sections with a blank line before the `assert` block.
- Database migrations: [Alembic](https://alembic.sqlalchemy.org/).

### Project structure

Backend services use a three-layer architecture:

- `<package>/api/` — HTTP layer: routes, schemas, middleware.
- `<package>/domain/` — business logic: entities, services, exceptions.
- `<package>/infrastructure/` — concrete implementations of domain interfaces. Test stubs (`stubs.py`) live here alongside the real adapters they replace.

Interfaces (ports) are defined in the module that depends on them, not in the module that implements them. This applies at every layer: domain interfaces live in `domain/`, and if one infrastructure component depends on another, the interface lives in the depending component's package.

Tests mirror the source structure:

```
tests/
  unit/
    api/
    domain/
    infrastructure/
  integration/
  fixtures/
```

## Go projects

- Linter: [golangci-lint](https://golangci-lint.run/).
- Testing: standard library `testing` package.

### Project structure

Applications follow the standard Go layout:

- `cmd/<name>/` — entry point for each binary (`main.go` lives here).
- `internal/` — all application code; not importable by external packages.
- No `pkg/` directory for applications — use `pkg/` only in libraries meant to be imported by others.
